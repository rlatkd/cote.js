# 0002. POC 범위 및 디자인 방향

- **상태**: Accepted
- **일자**: 2026-07-09

## 맥락

전체 스택을 한 번에 만들 수 없으므로, 먼저 프론트엔드 POC로 서비스의 얼굴을 만든다. "풀이 페이지 하나"만으로는 서비스처럼 보이지 않는다는 판단.

## 결정

- **사이트 구조**: 백준식 다중 페이지 — 홈(대시보드) / 문제 목록 / 문제 상세 / 채점 현황.
- **풀이 방식**: 리트코드식 **통합 split view** — 문제 상세 페이지 안에서 좌:지문 / 우:Monaco 에디터 + 실행 결과. (백준식 분리형 아님)
- **비주얼**: 백준의 낡은 외관 대신 현대적(미니멀·타이포·다크). 다크 모드 기본 + 라이트 토글.
- **컴포넌트**: Tailwind 직접(컴포넌트 라이브러리 없이).

## 근거

- 다중 페이지 구조라야 실제 서비스로 보이고 포트폴리오로 성립.
- 통합 split view는 사용성이 우수하고, 정적(지문)+인터랙션(에디터) 혼합이 Next RSC의 교과서적 활용 예시가 됨([0004]와 연결).
- 백준 구조는 익숙하되 외관은 차별화 → "구조는 검증된 것, 겉모습은 현대적".

## 검토한 대안

- 풀이 화면 단독(사이트 느낌 없음), 백준 클래식 외관 재현(낡아 보임 → 포트폴리오 마이너스), 리트코드식 다크 일변도(백준식 목록 구조 요구와 불일치), shadcn/ui·MUI(직접 Tailwind 선택으로 제외) — 미채택.

## 결과

- 프론트 POC 4개 페이지 구현 대상 확정.
- 통합 split view 때문에 문제 상세는 클라이언트 인터랙션 비중이 큼 → 아키텍처에서 client island 처리 필요.
